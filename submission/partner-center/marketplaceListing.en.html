Guest Sponsor Info for Microsoft Entra B2B is a SharePoint Online web part for organizations that use a dedicated guest landing page in their resource tenant, meaning the Microsoft 365 tenant that invited guests access. On that landing page, SharePoint can provide the surrounding orientation with welcome text, curated links, tenant-pinned Microsoft 365 entry points, and self-service account actions. Guest Sponsor Info provides the missing human context: it answers who can help me here? by showing the sponsor or backup sponsor assigned to the signed-in guest account and giving guests a clear contact path from day one.

This solution is intended for organizations that manage Microsoft Entra B2B guest onboarding through a dedicated identity and access management process and operate a SharePoint Online landing page for external users in that resource tenant. The primary end users are the guest users themselves, such as partners, suppliers, customers, and project guests. The web part helps those guests understand who their internal sponsor is, how to contact that person, and where to start in the tenant.

  • Shows assigned sponsors as configurable cards with profile photo or initials fallback, display name, and job title.
  • Opens a rich contact panel with email, work phone, mobile, office address, manager details, and optional map support.
  • Can show live Microsoft Teams presence and provides direct links for Microsoft Teams chat and audio call, plus email.
  • Supports full, compact, and automatic layouts for different SharePoint page zones.
  • Supports multiple sponsors, fallback sponsors, and filtering of unavailable accounts so guests keep seeing a meaningful contact path.
  • Includes preview mode for page authors and is available in 17 languages.

What this solution does and does not include

  • This solution is the sponsor-visibility component of a SharePoint guest landing page. The surrounding landing-page elements, such as welcome text, SharePoint Quick Links, tenant-pinned Microsoft 365 entry points, account self-service links, news, and audience-targeted page content, are configured separately with standard SharePoint features.
  • The solution does not create guest accounts, assign sponsors, add guests to Microsoft Teams, or run external access lifecycle workflows. It displays and uses sponsor data that already exists in your Microsoft 365 tenant.

Requirements

Requirements for the web part

  • Requires SharePoint Online and a guest-accessible landing page in the resource tenant that guests access.
  • Page editors always get a realistic preview in SharePoint edit mode, even without guest sign-in or live sponsor data.
  • The web part can also be tested on the published page in its built-in demo mode. The first-run setup wizard proposes this option by default and turns it on when selected, so sample sponsors can be shown on the live page without the Guest Sponsor API. After that initial setup, public demo mode can still be turned off or on again in the web part settings.
  • The app is delivered through the tenant App Catalog, but it must be added manually to the specific SharePoint site where you want to use it. It is not deployed automatically across the entire tenant. This usually fits the intended use on selected guest landing page sites.
  • For guest-facing use, administrators must also ensure that guest users can load the web part files. In typical marketplace-based deployments, the recommended approach is to enable the Office 365 Public CDN for the app files stored in the Tenant App Catalog. If that is not suitable for your environment, the setup guide below describes alternative rollout options.
  • On published pages, the web part is shown only to guest users by default. Internal users in that same resource tenant do not see it, by design, because the web part is intended to show sponsor contacts for guest accounts rather than manager information for employees. Public demo mode is a separate optional testing setting.

Additional requirements for live sponsor data in production

  • Showing real sponsor data in production requires Microsoft Entra B2B guest accounts with one or more sponsors already assigned, plus the separately provided, source-available Guest Sponsor API, which runs as an Azure Function in your Azure subscription.
  • The recommended production path is the built-in deployment wizard PowerShell script, which provides a guided, text-based deployment flow. The simplest option is a single PowerShell command in Azure Cloud Shell, so no local installation is required. Local setup on Windows, macOS, and Linux is also supported.
  • Deploying the Guest Sponsor API requires an Azure subscription and administrator permissions in Azure and Microsoft Entra. For the initial deployment, the Azure requirement is Owner permissions at subscription scope. On the Microsoft Entra side, typical setups require Cloud Application Administrator plus Privileged Role Administrator.
  • The web part itself does not request Microsoft Graph permissions. Graph data is read server-side through the Guest Sponsor API using its managed identity.
  • Live Microsoft Teams presence is optional and requires an additional Microsoft Graph permission on the Guest Sponsor API. Without it, sponsor cards are still shown, but without presence indicators.
  • Microsoft Teams chat and call links require the guest to already be added to at least one Microsoft Team in the resource tenant. If Microsoft Teams is not yet available for that guest, the solution shows a clear notice instead of a broken action.
  • Inline map preview requires an Azure Maps key. In the default Guest Sponsor API deployment path, an Azure Maps account is created for you, and you only need to copy its key into the web part settings. Without a key, the solution can still show an external map link.
